# 复用说明 · 框架可平移性

> 这套系统的价值不止于"诊断新客转化"这一个场景。它的内核是一个**可复用的跨域诊断框架**——
> 换一个锚点指标、换几张宽表，就能平移到完全不同的业务问题。本文说明框架抽象、扩展方法与可平移场景。

## 一、框架内核（一句话）

**锁定一个跨域锚点指标 → 在它身上对齐多域信号 → 找跨团队断点 → 用增长方法论生成排好序的行动清单。**

为什么这个内核可复用：增长卡点几乎总是掉在团队的缝里（A 团队看域 1、B 团队看域 2，没人同时看全）。
只要一个指标的解释天然需要跨多个团队的数据，这套框架就适用。

## 二、四层架构与解耦点

| 层 | 职责 | 复用时是否要改 |
|---|---|---|
| 语义层 `semantic.yaml` | 指标口径、维度、归因方法的单一事实来源 | **改**：换锚点指标与口径 |
| 数据层 `data/*.csv` | 每个域一张预聚合宽表，原始明细不出本地 | **换**：替换为真实导出，schema 对齐即可 |
| 归因引擎 `tools.py` / `analyze.py` | 跨域 join、基准、显著性、混杂守门、RICE、单位经济 | **基本不改**：加新域＝加一个查询函数 |
| 表达层 `SKILL.md` | LLM 在确定性数字上组织诊断叙事与打法 | 微调：结论体系按新场景措辞 |

**关键解耦**：架构与模型解耦（表达层可替换为任意内部模型）；原始数据与 LLM 解耦（只有聚合结果进 LLM，天然合规）。

## 三、新增一个域 = 三步（无需改主流程）

1. **加一张预聚合宽表**：放进 `data/`，JOIN 键统一为 `category + country`（或你的业务对象主键）。
   > 务必预聚合，不要导原始 event。
2. **在 `tools.py` 加一个查询函数**：仿照 `query_supply()` / `query_search_trend()`，返回聚合小结果。
3. **在跨域 join 处 merge 进来**：在 `supply_demand_mismatch()` 或 `external_context()` 里按主键并入新信号。

域越多，人脑越看不过来，这套系统越值钱——扩展性就是护城河。

## 四、换成真实数据（无需改代码）

把导出的 CSV 放进 `data/`，schema 与合成数据一致即可，换完直接重跑：

| 文件 | 粒度 | 关键列 |
|---|---|---|
| `behavior_funnel.csv` | date×channel×country×category×price_band | exposure_uv, click_uv, search_uv, addcart_uv, order_uv |
| `supply_depth.csv` | date×country×category | sku_count, in_stock_rate, oos_rate, price_competitiveness, sell_through_rate |
| `acquisition.csv`（可选） | date×channel×country | new_users, spend, cac |
| `search_trend.csv`（外部·可选） | date×country×category | search_index, wow_change |
| `event_calendar.csv`（外部·可选） | country | upcoming_event, days_to_event, is_peak_window |

## 五、可平移场景（换锚点指标即可）

| 场景 | 锚点指标 | 要对齐的域（示例） | 找的跨团队断点 |
|---|---|---|---|
| **新客增长**（本项目） | 新客首单转化率 | 行为×供给×搜索×大促 | 高意图新客撞上供给不足 |
| **复购/留存** | 30 日复购率 | 首单体验×物流时效×客服×商品质量 | 哪类首单体验毁掉了复购 |
| **大促冷启动** | 大促期 GMV 达成率 | 蓄水加购×库存备货×价格×流量分配 | 蓄水充足但备货/价格没跟上的品类 |
| **品类增长** | 品类渗透率 | 搜索需求×SKU 丰富度×履约×竞品价 | 有需求但供给/履约接不住的品类×地区 |
| **商家经营参谋** | 商家 GMV 健康分 | 流量×转化×库存×售后×评分 | 商家经营链路上拖累 GMV 的最弱环节 |

平移步骤固定：① 换 `semantic.yaml` 的锚点指标与口径 → ② 准备该场景的多域宽表 → ③ 调整 `SKILL.md` 结论措辞 → ④ 重跑。

## 六、复用时保留的"可信三件套"

无论平移到哪个场景，引擎的可信机制都应保留：
- **样本量门槛**（避免追噪声）
- **统计显著性检验 + 分组基准**（缺口是否真实）
- **混杂守门 + 归因置信度 + 验证设计**（区分"指标异常"与"异常归谁"，给的是排好序的强假设，不冒充因果）
